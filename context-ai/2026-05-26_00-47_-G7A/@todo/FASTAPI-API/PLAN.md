# FASTAPI-API — Plan

## Context

The CLI is the primary interface today, but ADR-002 commits us to a multi-interface architecture. A FastAPI layer will expose `src/services/` via HTTP REST, enabling the VS Code extension (and any future UI) to talk to context-ai without shelling out to the CLI.

**Hard prerequisite:** DI-REFACTOR (#10) must be complete first — the current singleton pattern is not thread-safe for concurrent HTTP requests.

## Goals

1. Create `src/api/` layer that mirrors CLI commands as REST endpoints
2. Streaming SSE support for `ask` and `chat` (same streaming the CLI already does)
3. Auth middleware (API key header) to protect endpoints
4. CORS configuration for local VS Code webview and future web UIs
5. Proper HTTP error responses mapped from existing exception hierarchy
6. OpenAPI docs auto-generated by FastAPI

## Scope

**In scope:**
- `src/api/` package: routers, middleware, schemas, dependencies
- REST endpoints for: generate, select, query, ask, chat, config, storage
- SSE streaming for AI responses
- API key auth (simple header-based, configurable)
- CORS middleware
- Error handler middleware mapping `ContextAIError` hierarchy to HTTP status codes
- Health check endpoint
- OpenAPI/Swagger docs (built-in FastAPI)

**Out of scope:**
- WebSocket support (SSE is sufficient for now)
- User management / OAuth / JWT (overkill for local tool)
- Database migrations (ChromaDB handles its own storage)
- Docker / deployment infra (separate concern)
- Rate limiting (local tool, not needed yet)

## Decisions

1. **FastAPI over Flask/Django** — async-native, auto OpenAPI docs, Pydantic integration already in our stack
2. **SSE over WebSocket** — simpler, HTTP-based, sufficient for streaming text responses
3. **API key auth** — stored in config, passed via `X-API-Key` header. Simple and sufficient for local use
4. **Router-per-command** — each CLI command gets a FastAPI router, mirrors existing `commands/` structure
5. **Pydantic v2 schemas** — already using Pydantic v2 in config layer, reuse patterns for request/response models
6. **DI via FastAPI Depends** — after DI-REFACTOR, inject services through FastAPI's dependency injection system

## Phases

### Phase 1: Project Setup & Foundation
- Add `fastapi` + `uvicorn` to dependencies (optional extra: `pip install context-ai-alpha[api]`)
- Create `src/api/` package structure: `__init__.py`, `app.py`, `dependencies.py`, `schemas/`, `routers/`, `middleware/`
- Create FastAPI app factory in `app.py`
- Health check endpoint (`GET /health`)
- Error handler middleware mapping exceptions to HTTP responses
- Console script entry point: `context-ai-api` → `uvicorn src.api.app:create_app`
- **Effort:** Small

### Phase 2: CRUD Endpoints (generate, select, config, storage)
- `POST /generate` — trigger embedding generation for a project
- `GET /collections` — list collections (maps to `storage list`)
- `DELETE /collections/{name}` — delete collection
- `GET /config` — read current config
- `PUT /config` — update config values
- `POST /select` — select files for context
- Pydantic request/response schemas for each
- **Effort:** Medium

### Phase 3: AI Endpoints with Streaming (query, ask, chat)
- `POST /query` — semantic search, returns ranked results (JSON)
- `POST /ask` — single question, returns SSE stream
- `POST /chat` — conversational, accepts message history, returns SSE stream
- SSE streaming adapter wrapping existing `AIService` streaming
- **Effort:** Medium-Large

### Phase 4: Auth & CORS Middleware
- API key middleware: validate `X-API-Key` header against config value
- CORS middleware: configurable allowed origins (default: localhost)
- Config entries for API key and CORS origins
- **Effort:** Small

### Phase 5: Documentation & Integration
- Verify OpenAPI spec is complete and accurate
- Add API usage examples to docs
- Update VS Code extension to optionally use HTTP API instead of CLI subprocess
- Update ADR-002 with "Implemented" status for FastAPI
- **Effort:** Small
